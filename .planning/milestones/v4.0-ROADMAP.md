# Milestone v4.0: Salesforce Merge Integration

**Status:** SHIPPED 2026-02-10
**Phases:** 17-19
**Total Plans:** 9

## Overview

Send merge instructions from Quantico to Salesforce so customers can use Quantico as a standalone dedup/merge tool that syncs results back to their Salesforce org.

## Phases

### Phase 17: Core Integration

**Goal**: Quantico can authenticate with Salesforce, generate merge instruction payloads, and deliver them to Salesforce staging object
**Depends on**: Phase 16 (v3.0 complete)
**Requirements**: SFI-01, SFI-02, SFI-03, SFI-04, SFI-05, SFI-06, SFI-07, SFI-08, SFI-09, SFI-10, SFI-11, SFI-12, SFI-13, SFI-14
**Plans:** 5 plans

Plans:
- [x] 17-01-PLAN.md -- Foundation: database schema, entity types, SFID prefixes, repository, encryption utility
- [x] 17-02-PLAN.md -- OAuth 2.0: service with token management, handler with config/connect endpoints
- [x] 17-03-PLAN.md -- Payload: merge instruction builder, batch assembler with unique IDs
- [x] 17-04-PLAN.md -- Delivery: async batch delivery service, queue/trigger/status endpoints
- [x] 17-05-PLAN.md -- Admin UI: integrations hub, Salesforce config page, connection flow, job table

**Success Criteria:**
1. Admin can configure Salesforce Connected App (Client ID, Client Secret, Redirect URL) in Quantico admin panel
2. Admin can authorize Quantico to access Salesforce via OAuth 2.0 authorization flow
3. Quantico generates valid merge instruction JSON from dedup resolution results
4. Quantico batches up to 200 merge instructions per API call with unique batch_id and instruction_ids
5. Quantico POSTs batched merge instructions to Salesforce staging object via REST API
6. Quantico proactively refreshes OAuth access tokens before expiry

### Phase 18: Rate Limiting & Error Handling

**Goal**: Quantico respects Salesforce API limits and handles errors intelligently
**Depends on**: Phase 17
**Requirements**: SFI-15, SFI-16, SFI-17, SFI-18, SFI-19
**Plans:** 2 plans — completed 2026-02-10

Plans:
- [x] 18-01-PLAN.md -- Foundation: API usage tracking migration, entity types, RateLimitService with sliding window
- [x] 18-02-PLAN.md -- Integration: exponential backoff in delivery, quota check, force flag, quota endpoint, main.go wiring

**Success Criteria:**
1. Quantico tracks API usage per org over 24-hour rolling windows
2. Quantico does not exceed 100,000 Salesforce API calls per 24-hour period per org
3. Quantico pauses batch delivery automatically at 80% API capacity
4. Quantico implements exponential backoff for 429 errors (5s, 10s, 20s, 40s with max 5 retries)
5. Admin can manually trigger merge delivery to override rate limiting pauses

### Phase 19: Audit Logging & Admin Configuration

**Goal**: Quantico logs all merge instruction delivery for compliance and provides admin UI for configuration and monitoring
**Depends on**: Phase 18
**Requirements**: SFI-20, SFI-21, SFI-22, SFI-23, SFI-24, SFI-25, SFI-26, SFI-27
**Plans:** 2 plans — completed 2026-02-10

Plans:
- [x] 19-01-PLAN.md -- Backend: Salesforce audit event types, delivery logging, batch_id/success filters, retention service
- [x] 19-02-PLAN.md -- Frontend: audit log Salesforce filters and descriptions, test connection button, admin controls

**Success Criteria:**
1. Quantico logs every merge instruction sent with batch_id, instruction_id, timestamp, delivery status
2. Audit logs stored with 7-year retention (SOX compliance) and tamper-evident hash chain
3. Admin can query audit logs by batch_id, date range, org, and result status via admin UI
4. Admin page shows Salesforce connection status and allows testing connection
5. Admin can enable/disable Salesforce sync for an org via toggle control
6. Admin can manually trigger immediate merge instruction delivery

## Milestone Summary

**Key Decisions:**
- Staging object pattern for Salesforce integration (external systems insert, Apex processes merges)
- OAuth 2.0 with proactive token refresh before 5-min expiry
- AES-256-GCM encryption for OAuth tokens with environment variable key storage
- Master DB for OAuth config, tenant DB for sync job history
- State-based CSRF protection for OAuth callback
- 15-to-18 character Salesforce ID checksum algorithm in Go
- Batch size limit of 200 instructions per batch (Salesforce Composite API limit)
- Async delivery with per-org concurrency control (max 1 concurrent per org)
- HTTP 202 pattern for immediate job ID return with background execution
- Idempotency key format {orgID}-{batchID} to prevent duplicate deliveries
- 24-hour sliding window for API usage tracking
- Exponential backoff: 5s initial, 2x multiplier, 40s max, 50% jitter, max 5 retries
- Force flag on ManualTrigger bypasses quota check for emergency overrides
- JSON LIKE search for batchId filter (avoids schema migration)
- 7-year audit log retention with VerifyChainSince for partial chain verification

**Issues Resolved:**
- Audit logs API route mismatch (frontend calling /admin/audit-logs, backend at /audit-logs) — pre-existing bug fixed during deployment

**Issues Deferred:**
- v4.1 features: delivery status dashboard, pre-flight validation, dead letter queue, field mapping presets, setup wizard, extensible integration framework

**Technical Debt Incurred:**
- Audit retention cleanup is a service with no scheduled execution (no cron yet — can be added as quick task)
- No Phase 17-05 SUMMARY.md (admin UI plan had no dedicated executor summary)
