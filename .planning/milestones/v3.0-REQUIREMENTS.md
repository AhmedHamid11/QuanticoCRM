# Requirements Archive: v3.0 Deduplication System

**Archived:** 2026-02-09
**Status:** SHIPPED

This is the archived requirements specification for v3.0.
For current requirements, see `.planning/REQUIREMENTS.md` (created for next milestone).

---

## Overview

Comprehensive, entity-agnostic deduplication system with scoring-based matching, import integration, manual merge with undo, and background scanning.

**User decisions:**
- Matching criteria: Full profile scoring (all available fields)
- Import behavior: Block with review (require user decision)
- Merge rules: Manual field selection per conflict
- Entity scope: Generic system for all entities

---

## v3.0 Requirements

### Detection & Matching

- [x] **DETECT-01**: Admin can configure matching rules per entity type via UI
- [x] **DETECT-02**: System detects duplicates on record creation with configurable response (warn/block modes)
- [x] **DETECT-03**: System uses Jaro-Winkler algorithm for fuzzy name matching (threshold 0.88)
- [x] **DETECT-04**: System uses exact match for email after normalization (lowercase, trim)
- [x] **DETECT-05**: System uses E.164 normalization for phone matching via libphonenumber
- [x] **DETECT-06**: System calculates confidence scores (0-100) using weighted field scoring
- [~] **DETECT-07**: System applies negative signals to reduce false positives — *Framework ready, specific logic not implemented*
- [x] **DETECT-08**: System uses SQL blocking strategy to limit candidate comparisons
- [x] **DETECT-09**: User sees tiered confidence levels in duplicate review

### Import Integration

- [x] **IMPORT-01**: CSV import runs duplicate detection during analyze step
- [x] **IMPORT-02**: Import preview shows potential duplicates with match scores
- [x] **IMPORT-03**: User can choose action for each duplicate row (skip/update/import/merge)
- [x] **IMPORT-04**: Import detects duplicates within the file (rows duplicating each other)
- [x] **IMPORT-05**: Import blocks proceeding until duplicate decisions are made

### Merge Capabilities

- [x] **MERGE-01**: User can merge two or more duplicate records into one
- [x] **MERGE-02**: User selects survivor record (which record ID to keep)
- [x] **MERGE-03**: User selects field values to keep via side-by-side UI
- [x] **MERGE-04**: Merge transfers all related records to survivor
- [x] **MERGE-05**: System discovers related records dynamically from entity metadata
- [x] **MERGE-06**: Merge executes as atomic transaction (all or nothing)
- [x] **MERGE-07**: Merge creates audit log entry with who/when/what
- [x] **MERGE-08**: System stores pre-merge snapshots for undo capability
- [x] **MERGE-09**: User can undo merge within 30 days
- [x] **MERGE-10**: Multi-record merge (3+) executes as sequential pair merges with grouped undo
- [x] **MERGE-11**: Merge preview shows before/after comparison, related counts, and data loss warnings

### Background Processing

- [x] **BACKGROUND-01**: Admin can schedule duplicate scan jobs per entity type
- [x] **BACKGROUND-02**: Background scan uses cursor-based chunking (500 records per chunk)
- [x] **BACKGROUND-03**: Background jobs track status (Pending/Running/Completed/Failed) with progress
- [x] **BACKGROUND-04**: Admin receives in-app notification when scan completes
- [x] **BACKGROUND-05**: Background jobs use per-tenant rate limiting (max 2 concurrent)
- [x] **BACKGROUND-06**: Failed jobs can be retried from last checkpoint

### Admin UI

- [x] **UI-01**: Admin can manage matching rules in Settings > Data Quality > Duplicate Rules
- [x] **UI-02**: Admin can view duplicate review queue grouped by entity, sorted by confidence
- [x] **UI-03**: Merge wizard guides user through survivor/field selection/preview/confirmation
- [x] **UI-04**: Admin can bulk merge multiple duplicate groups with progress indicator
- [x] **UI-05**: Admin can view merge history with undo option (within 30 days)
- [x] **UI-06**: Admin can view and manage scheduled scan jobs
