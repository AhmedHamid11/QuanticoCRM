# Milestone v3.0: Deduplication System

**Status:** SHIPPED 2026-02-09
**Phases:** 11-16
**Total Plans:** 22

## Overview

Comprehensive, entity-agnostic deduplication system with scoring-based matching, import integration, manual merge with undo, and background scanning. Built a complete data quality subsystem including real-time detection, background scanning, and a full admin UI for managing duplicate rules, reviewing detected duplicates, and performing merges.

## Phases

### Phase 11: Detection Foundation

**Goal**: Core deduplication infrastructure with similarity algorithms and configurable rules
**Depends on**: Phase 10 (v2.0 complete)
**Plans**: 3 plans

Plans:
- [x] 11-01: Database schema and entity types (matching_rules table, blocking index columns, Go types, repository)
- [x] 11-02: Similarity service (Jaro-Winkler fuzzy matching, E.164 phone normalization, weighted scorer)
- [x] 11-03: Blocking strategies, detection orchestrator, and admin API endpoints for rule management

**Details:**
- Created matching_rules table with configurable fields, logic operators, and fuzzy/exact modes
- Implemented Jaro-Winkler algorithm with 0.88 threshold for name matching
- Built weighted scoring system (email 40, phone 20, last name 20, first name 15, company 5)
- SQL blocking strategies: Soundex, prefix, domain, phone for candidate reduction
- Requirements: DETECT-01, DETECT-03, DETECT-04, DETECT-05, DETECT-06, DETECT-07 (partial), DETECT-08

### Phase 12: Real-Time Detection

**Goal**: Prevent new duplicates by detecting matches during record creation
**Depends on**: Phase 11
**Plans**: 4 plans

Plans:
- [x] 12-01: Pending alert infrastructure (migration, entity, repo, API endpoints)
- [x] 12-02: Async detection hooks (RealtimeChecker, GenericEntityHandler integration, 30s timeout goroutines)
- [x] 12-03: Frontend components (DuplicateAlertBanner, DuplicateWarningModal, confidence-tiered colors)
- [x] 12-04: Detail page integration (Contact, Account pages with DetailPageAlertWrapper)

**Details:**
- Async detection on record creation with optimistic save pattern
- Confidence tiers: High (>=95% red), Medium (>=85% yellow), Low (>=70% blue)
- Warn mode and block mode (block enforced at UI level)
- Requirements: DETECT-02, DETECT-09

### Phase 13: Merge Engine

**Goal**: Complete merge capability with field selection, related record transfer, audit logging, and undo
**Depends on**: Phase 11
**Plans**: 4 plans

Plans:
- [x] 13-01: Database schema, migrations, Go entity types, SFID prefix
- [x] 13-02: Merge snapshot repository, related record discovery service with completeness scoring
- [x] 13-03: Atomic merge execution service, snapshot-based undo, audit logging
- [x] 13-04: HTTP API handlers (preview, execute, undo, history) and route registration

**Details:**
- Atomic transaction: survivor update + related FK transfer + source delete
- 30-day undo via pre-merge snapshots
- Metadata-driven related record discovery (any entity with lookup pointing to merge target)
- Merge preview with side-by-side comparison and suggested survivor
- Requirements: MERGE-01 through MERGE-11

### Phase 14: Import Integration

**Goal**: Extend CSV import to detect and handle duplicates during import

### Phase 15: Background Scanning

**Goal**: Scheduled duplicate scans with job management, chunked processing, checkpoint recovery, and in-app notifications
**Depends on**: Phase 11, Phase 13
**Plans**: 3 plans

Plans:
- [x] 15-01: Database schema, entity types, SFID prefixes, and repository layer (multi-tenant routing)
- [x] 15-02: ScanJobService with 500-record chunked processing, checkpoint recovery, gocron v2 scheduling
- [x] 15-03: NotificationService, ScanJobHandler (15 endpoints), SSE progress streaming, main.go wiring

**Details:**
- Cursor-based chunking with checkpoint progress to avoid Turso 5-second timeout
- Per-tenant rate limiting (max 2 concurrent jobs)
- gocron v2 calendar-aware scheduling (daily/weekly/monthly)
- SSE progress streaming for real-time job status
- Requirements: BACKGROUND-01 through BACKGROUND-06

## Milestone Summary

**Key Decisions:**
- Single scrollable page instead of multi-step merge wizard (simpler UX)
- Side-by-side columns for field comparison in merge
- Auto-select survivor's field values or first non-empty as defaults
- Quick Merge uses merge preview's suggestedSurvivorId (completeness scoring)
- Bulk operations process sequentially with progress tracking
- SSE via EventSource with cleanup on navigation
- Schedule configuration via frequency presets (Daily/Weekly/Monthly) instead of cron
- ListAllPending sorts by highest_confidence DESC
- PaginatedResponse generic type for consistent pagination

**Issues Resolved:**
- Undo route typo fixed (/undo:snapshotId -> /undo/:snapshotId)
- Blocking key backfill for existing contacts on first dedup check
- Invalid blocking strategies handled gracefully
- Auto-provision dedup tables when missing from tenant DB
- Multiple browser-tested UI bugs fixed (navigation, display names, constraints, truncation)

**Issues Deferred:**
- DETECT-07: Negative signals framework ready but specific logic not implemented (can be added to scorer.go)
- IsBlockMode hardcoded to false in realtime.go (matching_rules table lacks block_mode column)

**Technical Debt Incurred:**
- Inline type definitions in duplicate-rules and scan-jobs pages (intentional for parallel execution)
- Block mode enforcement at UI level only, not backend
