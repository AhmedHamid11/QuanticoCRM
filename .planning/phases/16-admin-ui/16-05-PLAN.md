Create the scan job dashboard using Svelte 5 runes and Tailwind CSS. Uses existing backend APIs at `/scan-jobs/*`.

NOTE: If Plan 01 has not completed yet (data-quality.ts not available), import directly from `$lib/utils/api` and define types inline. If data-quality.ts exists, import from it.

**Page structure (per locked decisions):**

1. **Header:** "Scan Jobs" title, "Run Now" button (dropdown to select entity type, then triggers)

2. **Schedules table** (per user decision: "Simple status table: columns for entity type, schedule, last run, status, next run"):
   - Columns: Entity Type, Schedule, Last Run, Status, Next Run, Actions
   - **Entity Type:** e.g., "Contact", "Account"
   - **Schedule:** Human-readable, e.g., "Daily at 2:00 AM", "Weekly (Mon) at 3:00 AM", "Monthly (15th) at 1:00 AM"
   - **Last Run:** Relative time (e.g., "2 hours ago", "Yesterday") or "Never"
   - **Status:** Badge -- "Enabled" (green), "Disabled" (gray), "Running" (blue pulse)
   - **Next Run:** Date/time or "N/A" if disabled
   - **Actions:** Edit (pencil icon), Delete (trash icon), Toggle enable/disable
   - Sort by next run time (per user decision)

3. **Running scan progress** (per user decision: "Running scans show inline progress bar directly in the table row with percentage and records processed"):
   - When a job is running, the Status column shows an inline progress bar
   - Progress bar: Tailwind `bg-blue-500` bar inside `bg-gray-200` track, with percentage text
   - Text: "42% (1,050 / 2,500 records)"
   - Updated in real-time via SSE

4. **Schedule configuration** (per user decision: "Schedule configuration via preset dropdown options"):
   - Inline edit on row click or via edit button
   - Frequency: dropdown with "Daily", "Weekly", "Monthly"
   - If Weekly: day picker (Mon-Sun dropdown)
   - If Monthly: date picker (1-28 number input)
   - Hour and minute inputs (24h format)
   - Enable/disable toggle
   - Save button saves via PUT `/scan-jobs/schedules/{entityType}`

5. **New schedule:**
   - "Add Schedule" button
   - Entity type dropdown (filtered to entities without existing schedules)
   - Same config as edit: frequency, day, hour, minute
   - Save creates via PUT `/scan-jobs/schedules/{entityType}` (upsert)

6. **Recent job history** (below schedules table):
   - Title: "Recent Jobs"
   - Table: Date, Entity Type, Status (completed/failed/running), Records Processed, Duplicates Found, Duration, Actions
   - Failed jobs: show error message on hover/expand, "Retry" button
   - Per user decision: "Failed scans trigger in-app notification. Admin goes to dashboard to see details and manually trigger retry from last checkpoint"
   - Retry button calls POST `/scan-jobs/{id}/retry`
   - Pagination for job history

7. **Run Now flow:**
   - Click "Run Now" button
   - Dropdown or modal to select entity type
   - Calls POST `/scan-jobs/run` with entityType
   - Shows toast "Scan started for {entityType}"
   - New running job appears in table with progress bar

8. **SSE Progress Connection:**
   - On mount: connect EventSource to `/api/v1/admin/scan-jobs/progress/stream`
   - Use `PUBLIC_API_URL` or the same API_BASE from api.ts for the URL
   - Include auth token via URL query param (EventSource doesn't support headers) OR use cookie-based auth (credentials: include) -- since backend uses Fiber with cookie auth available, the existing session cookies should work
   - Listen for "progress" events:
     ```typescript
     eventSource.addEventListener('progress', (event) => {
       const data: ProgressEvent = JSON.parse(event.data);
       updateJobProgress(data);
     });
     ```
   - Update both the schedule table (inline progress) and job history table with progress
   - When status changes to "completed" or "failed", reload the tables
   - **CRITICAL:** Close EventSource in onMount cleanup to prevent memory leaks (per research Pitfall 5)
   ```typescript
   onMount(() => {
     const es = new EventSource(`${apiBase}/admin/scan-jobs/progress/stream`, { withCredentials: true });
     // ... event listeners
     return () => es.close();
   });
   ```
   - Note: EventSource auto-reconnects on error, no manual retry needed
