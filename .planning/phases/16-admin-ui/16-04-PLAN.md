<task type="auto">
  <name>Task 1: Build single-page merge wizard</name>
  <files>frontend/src/routes/admin/data-quality/merge/[groupId]/+page.svelte</files>
  <action>
Create the merge wizard page. The `groupId` param is the pending alert ID. On load, fetch the alert to get record IDs, then call merge preview.

**CRITICAL user decisions to honor:**
- Single scrollable page -- NO multi-step wizard, NO step navigation
- Side-by-side columns -- records in columns, fields in rows
- Click/radio to select which value to keep
- Differences highlighted
- Related records shown as full list, grouped by type, with source record indicated
- After merge: return to review queue, success toast with Undo link

**Page structure (single scrollable page, all sections visible):**

1. **Header:** "Merge Records" title, entity type label, "Back to Review Queue" link

2. **Section 1: Survivor Selection**
   - Radio buttons to select the surviving record
   - Each option shows: Record name, completeness score (as percentage bar), related record count
   - Pre-select the `suggestedSurvivorID` from preview
   - Changing survivor does NOT reset field selections (fields are independently chosen)

3. **Section 2: Field Comparison** (the core of the wizard)
   - Table structure: field labels in first column, then one column per record
   - Each field row:
     - Field label (from preview.fields)
     - Value from Record A, Value from Record B (etc.)
     - Radio button under each value to select which to keep
     - If values differ: highlight the row with a subtle yellow background (`bg-yellow-50`)
     - If values are identical: show normally, auto-select survivor's value
     - If one value is empty and other has data: auto-select the non-empty one, still allow override
   - Skip system fields (id, org_id, created_at, updated_at, archived_at, archived_by, merge_survivor_id)
   - Group fields logically (if field definitions have groups/sections, use them; otherwise flat list)

4. **Section 3: Related Records**
   - Show `relatedRecordCounts` from preview grouped by entity type
   - For each related entity type:
     - Header: "{EntityLabel} ({count} records)" with source record indicated
     - If records are included in preview, show them as a compact list
     - Note: "These records will be transferred to the surviving record"
   - If no related records: "No related records found"

5. **Section 4: Confirmation**
   - Summary: "Merging {N} records into {survivorName}. {M} related records will be transferred."
   - Warning if any field data will be lost (non-selected values from non-survivor records)
   - "Merge Records" primary button (blue, full width)
   - "Cancel" secondary button (returns to review queue)
   - Loading state during merge execution

6. **Post-merge behavior:**
   - On success: `goto('/admin/data-quality/review-queue')`
   - Show toast: "Records merged successfully. [Undo]" where Undo calls mergeUndo(snapshotId) and shows another toast on success
   - The undo link in the toast should be clickable and call the undo API

**State management:**
```typescript
let preview = $state<MergePreview | null>(null);
let loading = $state(true);
let error = $state<string | null>(null);
let survivorId = $state<string>('');
let fieldSelections = $state<Record<string, string>>({}); // fieldName -> recordId (which record's value to keep)
let merging = $state(false);
```

**On mount:**
1. Fetch the pending alert by navigating via the groupId (the alertId)
   - GET `/dedup/{entityType}/{recordId}/pending-alert` -- but we need entityType and recordId
   - Alternative approach: the groupId could encode `entityType:recordId` or the review queue could pass params via URL query
   - Better: accept query params `?entityType=Contact&recordIds=id1,id2,id3` from the review queue navigation
   - The review queue's "Merge" button navigates: `goto(`/admin/data-quality/merge/${alertId}?entityType=${alert.entityType}&recordIds=${recordId},${matchIds}`)`
2. Parse entityType and recordIds from URL query params
3. Call POST `/merge/preview` with { recordIds, entityType }
4. Initialize survivorId to `suggestedSurvivorID`
5. Auto-select field values: for each field, pick survivor's value unless empty (then pick first non-empty)

**Build merge request on submit:**
```typescript
const mergedFields: Record<string, unknown> = {};
for (const [fieldName, selectedRecordId] of Object.entries(fieldSelections)) {
  const record = preview.records.find(r => r.id === selectedRecordId);
  if (record) mergedFields[fieldName] = record[fieldName];
}
const req: MergeRequest = {
  survivorId,
  duplicateIds: preview.records.filter(r => r.id !== survivorId).map(r => r.id),
  mergedFields,
  entityType
};
```

Create the merge history page showing recent merges with undo capability.

**Page structure (per user decisions):**

1. **Header:** "Merge History" title, entity type filter dropdown

2. **History table:**
   - Columns: Date, Entity Type, Survivor ID (clickable link to record), Duplicates Merged (count), Merged By, Status, Actions
   - Status column shows either "Active" (can undo) or "Permanent" (expired or already undone)
   - Actions column: "Undo" button for entries where `canUndo` is true
   - Undo button disabled with tooltip "Expired" if past 30-day window (compare expiresAt)
   - Sort by date descending (most recent first)
   - Pagination at bottom

3. **Undo flow:**
   - Click "Undo" button
   - Confirm dialog: "This will restore the merged records to their pre-merge state. Continue?"
   - Call POST `/merge/undo/{snapshotId}`
   - On success: toast "Merge undone successfully", update row to show "Undone" status, disable undo button
   - On error: toast with error message (e.g., "Undo window expired" if 30 days passed between load and click)

4. **Empty state:** "No merge operations recorded yet."

**State management:**
