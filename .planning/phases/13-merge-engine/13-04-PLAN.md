**Endpoint 1: POST /merge/preview — Preview(c *fiber.Ctx) error**

Request body: `entity.MergePreviewRequest` (recordIds: []string, entityType: string)

Response: `entity.MergePreview` (records, suggestedSurvivorId, completenessScores, relatedRecordCounts, fields)

Logic:
1. Parse request body, validate at least 2 recordIds
2. Get orgID and userID from c.Locals
3. Verify entity exists: `metadataRepo.GetEntity(ctx, orgID, entityType)`
4. Get field definitions: `metadataRepo.ListFields(ctx, orgID, entityType)`
5. Fetch each record: `util.FetchRecordAsMap(ctx, db, tableName, recordID, orgID)` — return 404 if any record not found
6. Calculate completeness scores for each record
7. Suggest survivor (most complete)
8. Count related records for each record: `discoveryService.CountRelatedRecords(ctx, db, orgID, entityType, recordIDs)`
9. Return MergePreview

**Endpoint 2: POST /merge/execute — Execute(c *fiber.Ctx) error**

Request body: `entity.MergeRequest` (survivorId, duplicateIds, mergedFields, entityType)

Response: `entity.MergeResult` (survivorId, snapshotId, mergedAt)

Logic:
1. Parse request body, validate survivorId, at least 1 duplicateId, entityType
2. Get orgID, userID from c.Locals
3. Call `mergeService.ExecuteMerge(c.Context(), db, orgID, userID, req)`
4. Return 200 with MergeResult on success
5. Return 500 with error on failure

Per user decision on multi-record merge (3+ records): "Pair-by-pair merging, not all-at-once. User merges two records first, then merges the result with the next duplicate."
- The backend only needs to handle 1 survivor + N duplicates per call
- The frontend will call this endpoint multiple times for 3+ record merges
- No special multi-record logic needed in the backend (per user decision)

**Endpoint 3: POST /merge/undo/:snapshotId — Undo(c *fiber.Ctx) error**

URL param: snapshotId

Response: 200 `{"message": "Merge undone successfully", "survivorId": "..."}` or error

Logic:
1. Get snapshotId from c.Params
2. Get orgID, userID from c.Locals
3. Call `mergeService.UndoMerge(c.Context(), db, orgID, userID, snapshotID)`
4. Return 200 on success, 400/404/500 on error

Error cases:
- Snapshot not found: 404
- Snapshot expired: 400 "Merge cannot be undone after 30 days"
- Snapshot already consumed: 400 "Merge has already been undone"

**Endpoint 4: GET /merge/history — History(c *fiber.Ctx) error**

Query params: page (default 1), pageSize (default 20), entityType (optional filter)

Response: `{"data": [...MergeHistoryEntry], "total": N, "page": N, "pageSize": N}`

Logic:
1. Parse query params
2. Get orgID from c.Locals
3. Call `mergeRepo.ListByOrg(ctx, orgID, page, pageSize)`
4. Map MergeSnapshot to MergeHistoryEntry (compute canUndo: not expired AND not consumed)
5. Parse duplicateIDs from JSON for each entry
6. If entityType filter provided, only return matching entries

**RegisterRoutes(app fiber.Router):**
```go
func (h *MergeHandler) RegisterRoutes(app fiber.Router) {
    merge := app.Group("/merge")
    merge.Post("/preview", h.Preview)
    merge.Post("/execute", h.Execute)
    merge.Post("/undo/:snapshotId", h.Undo)
    merge.Get("/history", h.History)
}
```

All endpoints are on the `protected` route group (authenticated users, not admin-only) since any user who can view records should be able to merge duplicates they can see. The merge wizard UI (Phase 16) will handle role-based access if needed.
