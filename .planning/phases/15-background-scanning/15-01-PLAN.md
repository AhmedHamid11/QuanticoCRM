Create four SQL migration files in backend/internal/migrations/:

**057_create_scan_schedules.sql:**
```sql
CREATE TABLE IF NOT EXISTS scan_schedules (
    id TEXT PRIMARY KEY,
    org_id TEXT NOT NULL,
    entity_type TEXT NOT NULL,
    frequency TEXT NOT NULL,           -- 'daily', 'weekly', 'monthly'
    day_of_week INTEGER,               -- 0=Sunday..6=Saturday (for weekly)
    day_of_month INTEGER,              -- 1-28 (for monthly, cap at 28 to avoid edge cases)
    hour INTEGER NOT NULL DEFAULT 3,   -- 0-23, hour of day (UTC)
    minute INTEGER NOT NULL DEFAULT 0, -- 0-59
    is_enabled INTEGER NOT NULL DEFAULT 1,
    last_run_at TEXT,                  -- ISO timestamp of last execution
    next_run_at TEXT,                  -- ISO timestamp of next scheduled run
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(org_id, entity_type)        -- One schedule per entity per org
);
CREATE INDEX IF NOT EXISTS idx_scan_schedules_org ON scan_schedules(org_id, is_enabled);
CREATE INDEX IF NOT EXISTS idx_scan_schedules_next_run ON scan_schedules(next_run_at, is_enabled);
```

**058_create_scan_jobs.sql:**
```sql
CREATE TABLE IF NOT EXISTS scan_jobs (
    id TEXT PRIMARY KEY,
    org_id TEXT NOT NULL,
    entity_type TEXT NOT NULL,
    schedule_id TEXT,                  -- NULL for manual "Run Now" jobs
    status TEXT NOT NULL DEFAULT 'pending',  -- 'pending', 'running', 'completed', 'failed', 'cancelled'
    trigger_type TEXT NOT NULL DEFAULT 'scheduled', -- 'scheduled', 'manual'
    total_records INTEGER NOT NULL DEFAULT 0,
    processed_records INTEGER NOT NULL DEFAULT 0,
    duplicates_found INTEGER NOT NULL DEFAULT 0,
    error_message TEXT,
    started_at TEXT,
    completed_at TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_scan_jobs_org_status ON scan_jobs(org_id, status);
CREATE INDEX IF NOT EXISTS idx_scan_jobs_org_entity ON scan_jobs(org_id, entity_type, created_at);
CREATE INDEX IF NOT EXISTS idx_scan_jobs_schedule ON scan_jobs(schedule_id, created_at);
```

**059_create_scan_checkpoints.sql:**
```sql
CREATE TABLE IF NOT EXISTS scan_checkpoints (
    id TEXT PRIMARY KEY,
    job_id TEXT NOT NULL,
    last_offset INTEGER NOT NULL DEFAULT 0,
    last_processed_id TEXT,            -- Last record ID processed (for cursor-based resume)
    retry_count INTEGER NOT NULL DEFAULT 0,
    chunk_size INTEGER NOT NULL DEFAULT 500,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(job_id)
);
```

**060_create_notifications.sql:**
```sql
CREATE TABLE IF NOT EXISTS notifications (
    id TEXT PRIMARY KEY,
    org_id TEXT NOT NULL,
    user_id TEXT NOT NULL,
    type TEXT NOT NULL,                -- 'scan_complete', 'scan_failed'
    title TEXT NOT NULL,
    message TEXT NOT NULL,
    link_url TEXT,                     -- URL to navigate when clicked
    is_read INTEGER NOT NULL DEFAULT 0,
    is_dismissed INTEGER NOT NULL DEFAULT 0,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    expires_at TEXT                    -- Auto-dismiss after 30 days
);
CREATE INDEX IF NOT EXISTS idx_notifications_user ON notifications(org_id, user_id, is_dismissed, created_at);
CREATE INDEX IF NOT EXISTS idx_notifications_expiry ON notifications(expires_at);
```

All migrations use IF NOT EXISTS for safety. Follow the same pattern as existing migrations (055, 056).

**4. Create backend/internal/repo/scan_job.go:**

Follow the pattern from repo/pending_alert.go (WithDB pattern for multi-tenant). Create `ScanJobRepo` with:

Schedule operations:
- `GetSchedule(ctx, orgID, entityType) (*entity.ScanSchedule, error)` - get schedule for entity
- `ListSchedules(ctx, orgID) ([]entity.ScanSchedule, error)` - all schedules for org
- `ListAllEnabledSchedules(ctx) ([]entity.ScanSchedule, error)` - across all orgs (for scheduler startup)
- `UpsertSchedule(ctx, schedule *entity.ScanSchedule) error` - INSERT OR REPLACE
- `DeleteSchedule(ctx, orgID, entityType string) error`
- `UpdateNextRunAt(ctx, scheduleID string, nextRun time.Time) error`

Job operations:
- `CreateJob(ctx, job *entity.ScanJob) error`
- `GetJob(ctx, jobID string) (*entity.ScanJob, error)`
- `ListJobs(ctx, orgID string, limit, offset int) ([]entity.ScanJob, int, error)` - with total count for pagination
- `ListJobsByEntity(ctx, orgID, entityType string, limit, offset int) ([]entity.ScanJob, int, error)`
- `GetRunningJobForEntity(ctx, orgID, entityType string) (*entity.ScanJob, error)` - check if already running
- `UpdateJobStatus(ctx, jobID, status string) error`
- `UpdateJobProgress(ctx, jobID string, processed, duplicates int) error`
- `UpdateJobCompletion(ctx, jobID, status string, totalRecords, processed, duplicates int) error`
- `CountRunningJobsForOrg(ctx, orgID string) (int, error)` - for per-tenant rate limiting (max 2)

Checkpoint operations:
- `SaveCheckpoint(ctx, checkpoint *entity.ScanCheckpoint) error` - INSERT OR REPLACE
- `GetCheckpoint(ctx, jobID string) (*entity.ScanCheckpoint, error)`
- `IncrementRetryCount(ctx, jobID string) error`
- `DeleteCheckpoint(ctx, jobID string) error`

The repo takes `db.DBConn` (masterDBConn) and uses `WithDB(conn)` pattern for tenant database access. Schedule and notification queries run against master DB. Job and checkpoint queries run against tenant DB (since scan_jobs/scan_checkpoints are per-org data).

IMPORTANT: Since this is a multi-tenant system where each org may have its own Turso database, the migrations for scan_jobs, scan_checkpoints, and notifications need to run on EACH tenant database. The scan_schedules table goes on the MASTER database (since the scheduler needs to read all schedules at startup). Structure the repo methods accordingly:
- Schedule methods: use master DB connection directly
- Job/Checkpoint methods: use WithDB pattern (tenant DB)
- Notification methods: use WithDB pattern (tenant DB)
