```go
type SFDeliveryService struct {
    oauthService   *SalesforceOAuthService
    payloadBuilder *MergeInstructionBuilder
    batchAssembler *BatchAssembler
    repo           *repo.SalesforceRepo
    dbManager      *db.Manager  // for tenant DB routing

    runningJobs map[string]bool
    runningMu   sync.Mutex
}

func NewSFDeliveryService(
    oauthService *SalesforceOAuthService,
    payloadBuilder *MergeInstructionBuilder,
    batchAssembler *BatchAssembler,
    repo *repo.SalesforceRepo,
    dbManager *db.Manager,
) *SFDeliveryService
```

**Core methods:**

1. `QueueMergeInstructions(ctx, orgID string, inputs []MergeInstructionInput) (string, error)`:
   - Check connection status (must be "connected")
   - Build instructions via payloadBuilder.BuildInstructions
   - Get Salesforce org ID from connection's instance_url or stored config
   - Assemble batches via batchAssembler.AssembleBatches
   - For each batch: create a SyncJob record with status "pending" in tenant DB
     - Job ID: sfid.NewSyncJob()
     - BatchID: from assembled batch
     - IdempotencyKey: fmt.Sprintf("%s-%s", orgID, batchID)
     - BatchPayload: serialized JSON stored for retry
     - TotalInstructions: len(batch.MergeInstructions)
   - Launch async goroutine: `go s.executeBatchDelivery(orgID, jobIDs)`
   - Return first job ID (or batch summary) immediately (HTTP 202 pattern)

2. `executeBatchDelivery(orgID string, jobIDs []string)`:
   - Acquire per-org lock (only 1 concurrent delivery per org)
   - For each job:
     - Load job from repo
     - Update status to "running", set started_at
     - Get authenticated HTTP client from oauthService
     - Parse batch payload from job.BatchPayload
     - Call deliverBatch(ctx, orgID, client, batchPayload, job)
     - On success: update status to "completed", set completed_at, delivered count
     - On failure: update status to "failed", set error_message
   - Release per-org lock

3. `deliverBatch(ctx, orgID string, client *http.Client, payload []byte, job *entity.SyncJob) error`:
   - Get connection for instance_url
   - Build Salesforce REST API URL:
     ```
     {instanceURL}/services/data/v60.0/composite/sobjects
     ```
     Use env var `SALESFORCE_API_VERSION` (default "v60.0") for version.
   - Create HTTP request:
     ```go
     req, _ := http.NewRequestWithContext(ctx, "POST", url, bytes.NewReader(payload))
     req.Header.Set("Content-Type", "application/json")
     req.Header.Set("X-Idempotency-Key", job.IdempotencyKey)
     ```
     Note: Bearer token is handled by oauth2 HTTP client automatically.
   - Execute request via client.Do(req)
   - Check response status:
     - 200/201: Success - parse response for record IDs
     - 400: Bad Request - permanent error, extract Salesforce error message from response body
     - 401: Unauthorized - token issue, attempt one refresh then retry
     - 429: Rate limit - DO NOT retry here (Phase 18 handles advanced retry)
     - 500/502/503: Server error - basic retry (max 2 attempts with 2s delay)
   - Parse Salesforce error response: `[{"message": "...", "errorCode": "...", "fields": [...]}]`
   - Return error with Salesforce error details included

4. `GetJobStatus(ctx, orgID, jobID string) (*entity.SyncJob, error)`:
   - Load job from tenant DB via repo

5. `ListJobs(ctx, orgID string, limit, offset int) ([]entity.SyncJob, int, error)`:
   - List jobs from tenant DB via repo

6. `RetryJob(ctx, orgID, jobID string) error`:
   - Load job from repo
   - Verify status is "failed" (can only retry failed jobs)
   - Reset status to "pending", clear error_message, increment retry_count
   - Launch async: `go s.executeBatchDelivery(orgID, []string{jobID})`

**Error classification helper:**
```go
func isRetryableHTTPError(statusCode int) bool {
    switch statusCode {
    case 500, 502, 503, 504:
        return true
    default:
        return false
    }
}
```

**Basic retry for Phase 17** (Phase 18 adds exponential backoff):
- Max 2 retries for 5xx errors only
- 2-second fixed delay between retries
- 400/401/403/404: permanent failures, no retry
- 429: recorded as error (Phase 18 adds sophisticated handling)

IMPORTANT: Follow the scan_job async goroutine pattern from v3.0 (proven pattern). Use sync.Mutex for per-org concurrency control (same as ScanJobService.runningJobs pattern).
